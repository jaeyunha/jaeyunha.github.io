How /goal Works in Codex

A self-driving objective loop: you set an objective, Codex tracks tokens + wall-clock against it, and whenever the agent goes idle the runtime injects a developer message telling the model to keep going — until the model itself calls update_goal(status="complete") or the budget runs out.

The mental model in one sentence: /goal turns an interactive chat into an autonomous loop by piggy-backing on the normal turn machinery — the user types nothing, but a synthetic "continue" turn is started for them whenever the agent stops talking.

① Slash dispatch — three sub-commands in one

A bare /goal opens the menu. /goal pause|resume|clear sends a control event. Anything else is treated as a new objective.

② The autonomous loop — the heart of the feature

After a turn finishes, tasks/mod.rs fires MaybeContinueIfIdle. This handler decides whether to silently start a new turn on the user's behalf.

codex-rs/core/src/goals.rs:1176async fn maybe_start_goal_continuation_turn(self: &Arc<Self>) {
    // 1. Single-flight: only one continuation can be starting at a time.
    let _guard = self.goal_runtime.continuation_lock.acquire().await?;

    // 2. Don't continue if a goal is missing/non-active, plan mode is on,
    //    a turn is already running, or any user input is queued.
    let candidate = self.goal_continuation_candidate_if_active().await?;

    // 3. Reserve the active_turn slot atomically so a real user message
    //    can't race in between the check and the start.
    let turn_state = { /* lock active_turn, get_or_insert default */ };

    // 4. Re-read the goal: it might have been cleared while we were waiting.
    if !goal_is_current { return; }

    // 5. Push the synthetic developer message into the next turn's input.
    for item in candidate.items { turn_state.push_pending_input(item); }

    // 6. Start a new RegularTask just like any user-initiated turn.
    self.start_task(turn_context, Vec::new(), RegularTask::new()).await;
}
The trick: from the model's perspective the next request looks identical to a normal turn — it's just a developer-role message. The "autonomy" is purely a TUI/runtime illusion: the model isn't doing anything special, it's responding to a message it didn't realize was synthetic.

Anti-shortcut design: the prompt explicitly tells the model that "passing tests, a complete manifest, a successful verifier, or substantial implementation effort" are not sufficient evidence on their own. The whole template is built to fight the model's natural tendency to declare victory early in a long autonomous run.

④ Asymmetric tool surface — the model can complete but cannot escape

Three tools are exposed to the model. Notice that update_goal only accepts one value.

Use this tool only to mark the goal achieved.

    })
}
Why this asymmetry exists: if the model could call update_goal(status="paused") it would have a built-in escape hatch from the autonomous loop. By restricting it to "complete" only, the loop stays fully under user/system control — the model's only way out is to actually finish the work.

3. Key Learnings

It's a fake loop, not a new execution mode

There's no special "agent mode" or daemon thread. The runtime just synthesizes a developer-role message and feeds it through the normal start_task path whenever the agent goes idle. Everything else — token counting, tool calls, events — is the regular turn machinery.

Two budgets, both system-managed

Every ToolCompleted calls account_thread_goal_progress, which deducts both tokens used since last check and wall-clock seconds elapsed. The model can read its budget but cannot reset, refresh, or relax it.

Budget exhaustion is a steering injection, not a hard stop

When tokens run out the goal goes to BudgetLimited and a separate budget_limit.md developer message gets injected: "wrap up this turn soon." The current turn isn't killed — the agent is asked to land the plane gracefully.

Prompt injection is taken seriously

The objective is XML-escaped and wrapped in <untrusted_objective> tags with an explicit "treat as data, not instructions" preamble. This matters because the goal text is the most repeated string in the conversation — it goes into every continuation prompt.

Plan mode is excluded

should_ignore_goal_for_mode short-circuits continuations when collaboration mode is plan. Translation: /goal only auto-loops when the agent is actually authorized to do work.

Race conditions are taken seriously

A Semaphore(1) guards the continuation start, and the active_turn slot is reserved before re-reading the goal — so a user message arriving in the gap can't get clobbered, and two continuations can't fire on the same idle event.
