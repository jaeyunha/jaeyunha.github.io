How ooo auto Works in Ouroboros

A bounded state machine that takes a fuzzy goal and walks it through 7 phases — Socratic interview → A-grade Seed spec → execution handoff → optional Ralph QA loop — with a hard top-level deadline, persistent resume across process restarts, and a deterministic anti-self-rationalization gate.

Mental model in one sentence: if Codex's /goal is "keep poking the model until it says done," ooo auto is "drive a fuzzy idea through a typed pipeline of phases, each with its own timeout, where the model can never short-circuit a phase by claiming completeness."

① The phase enum — closed set, no improvisation

Every state the pipeline can be in is enumerated here. There are exactly 10 — three of them terminal. This closed set is what makes resume safe: a session in any state can be re-loaded and either continued or marked terminal.

src/ouroboros/auto/state.py:15class AutoPhase(StrEnum):
    """Closed set of phases for auto-mode resume and stall handling."""
    CREATED         = "created"
    INTERVIEW       = "interview"
    SEED_GENERATION = "seed_generation"
    REVIEW          = "review"
    REPAIR          = "repair"
    RUN             = "run"
    RALPH_HANDOFF   = "ralph_handoff"
    COMPLETE        = "complete"
    BLOCKED         = "blocked"
    FAILED          = "failed"

DEFAULT_TIMEOUT_SECONDS_BY_PHASE: dict[str, int] = {
    AutoPhase.INTERVIEW.value:       120,    # per-round, not total
    AutoPhase.SEED_GENERATION.value: 120,
    AutoPhase.REVIEW.value:           90,
    AutoPhase.REPAIR.value:           90,
    AutoPhase.RUN.value:              60,
}

DEFAULT_PIPELINE_TIMEOUT_SECONDS: float = 7200.0  # top-level 2h cap
Why it matters: a phase enum + per-phase timeout + top-level deadline is the entire safety story. The model never gets to invent a new state, can't stall a phase past its budget, and can't hold the whole pipeline past 2h. Compare this to /goal's loose "keep going until the model says done."

② The pipeline driver — phase machine, not an event loop

The 1500-line AutoPipeline.run() is one big sequential phase walk. Each phase is wrapped in asyncio.wait_for with a deadline-capped timeout. Failure inside a phase does not throw — it transitions to BLOCKED with a tool-name attribution.

src/ouroboros/auto/pipeline.py:179 (condensed)async def run(self, state: AutoPipelineState) -> AutoPipelineResult:
    # Top-level deadline check on resume — if 2h already elapsed, fail fast.
    if state.is_deadline_expired():
        state.mark_blocked(_RESUME_EXPIRED_MESSAGE, tool_name=PIPELINE_DEADLINE_TOOL_NAME)
        return self._result(state, ledger, blocker=state.last_error)

    # INTERVIEW phase, bounded by remaining deadline + per-phase timeout
    if state.phase in {AutoPhase.CREATED, AutoPhase.INTERVIEW}:
        interview_timeout = self._deadline_capped_timeout(state, INTERVIEW_TIMEOUT)
        try:
            interview = await asyncio.wait_for(
                self.interview_driver.run(state, ledger),
                timeout=interview_timeout,
            )
        except TimeoutError:
            state.mark_blocked(f"interview phase exceeded {INTERVIEW_TIMEOUT:.0f}s", ...)
            return self._result(state, ledger, blocker=state.last_error)
        state.transition(AutoPhase.SEED_GENERATION, "generating Seed from auto interview")

    # SEED_GENERATION → REVIEW (with deterministic ambiguity floor)
    if state.phase == AutoPhase.SEED_GENERATION:
        seed = await asyncio.wait_for(self.seed_generator(state.interview_session_id), ...)
        floor = deterministic_floor(ledger)        # anti-rationalization gate
        if floor > seed.metadata.ambiguity_score:
            seed = seed.model_copy(update={"metadata": ...{"ambiguity_score": floor}})

    # REVIEW ⇄ REPAIR loop until A-grade or budget
    if state.phase == AutoPhase.REVIEW:
        seed, review, repairs = await asyncio.wait_for(
            asyncio.to_thread(repairer.converge, seed, ledger=ledger, cancel_event=cancel_event),
            timeout=bounded_repair_timeout,
        )
        if not _grade_meets_required(review.grade_result.grade.value, state.required_grade):
            state.mark_blocked(f"Seed grade ... did not meet required grade ...", ...)

    # RUN → optional RALPH_HANDOFF chain
    if state.phase == AutoPhase.RUN:
        if self.complete_product and self.ralph_starter is not None:
            return await self._handoff_to_ralph(state, ledger, seed, review, run_subagent=None)
        state.transition(AutoPhase.COMPLETE, "execution started; using persisted run handle")
The trick: every phase is a guarded transition, not a callback. The pipeline can be killed mid-phase, restarted, and pick up where it left off — because the only durable thing is state.phase, not "where we are in some Python coroutine stack."

③ The deterministic ambiguity floor — sealing self-rationalization

When the LLM generates a Seed, it self-rates its own ambiguity. deterministic_floor() measures the same thing objectively from the ledger (open gaps, conflicting entries, assumption-only sections) and forces the score to at least that. The model can rate itself worse but not better.

src/ouroboros/auto/pipeline.py:394# Apply deterministic floor: the LLM-derived ambiguity_score
# cannot fall below what code can objectively measure from the
# ledger (open gaps, conflicting entries, assumption-only
# sections). Seals self-rationalization at the A-grade gate.
floor = deterministic_floor(ledger)
if floor > seed.metadata.ambiguity_score:
    seed = seed.model_copy(
        update={"metadata": seed.metadata.model_copy(
            update={"ambiguity_score": floor}
        )},
    )
Why this exists: the natural attack on a "we're done when grade ≥ A" pipeline is for the model to under-report its own ambiguity. The deterministic floor takes the easy lever away — code computes a minimum bound that the model literally cannot lie below. It's the structural equivalent of Codex's "update_goal can only set status=complete" trick: remove the dial the model would abuse.

④ Durable session intent — the operator's choice survives crashes

If you start with --complete-product and the process crashes after RUN, the resumed pipeline still chains to RALPH. Lowering is intentionally one-way — explicit True raises the bound, absence keeps the persisted truth.

src/ouroboros/auto/pipeline.py:196# Q00/ouroboros#773 (review-3): ``complete_product`` is durable session
# intent, not a per-invocation flag. On a fresh session the constructor
# writes the operator's choice into the state; on resume we honor the
# persisted value even when the caller forgot to re-pass the flag, so
# a session originally started with ``--complete-product`` keeps
# chaining RUN → RALPH_HANDOFF after restart. Lowering is intentional:
# explicit ``complete_product=True`` raises the bound; absence keeps
# the persisted truth.
if self.complete_product and not state.complete_product:
    state.complete_product = True
elif state.complete_product and not self.complete_product:
    self.complete_product = True
The principle: intents that affect safety scope (am I allowed to chain to a longer-running loop?) are sticky and one-way. You can opt up but not silently opt down on resume. Same idea as a permission cache that only grows.

4. Key Learnings

State machines beat "agent loops" for trust

The 10-value AutoPhase enum is the whole reason this is debuggable. You can grep state files for phase=BLOCKED and immediately know what stage failed and why (last_tool_name, last_error). A free-form agent loop has no such resolution.

Two-axis budgeting

Per-phase timeouts (120s for interview rounds, 60s for run handoff) plus a top-level 2h deadline. Each phase computes min(phase_timeout, remaining_deadline) via _deadline_capped_timeout, so no single slow phase can eat the whole budget.

Deterministic floors over prompt warnings

Both /goal and ooo auto fight self-rationalization. /goal uses prose ("don't accept proxy signals"). ooo auto uses code: deterministic_floor(ledger) measures objectively and overrides the LLM's self-grade. The code version is structurally stronger — the model can't argue against it.

BLOCKED is a feature, not a failure

state.recover() takes a BLOCKED session, looks at last_tool_name, finds the recoverable phase via _recoverable_phase_for_tool, and continues. Crashes, timeouts, grade misses — all leave a resumable state. FAILED is reserved for genuinely non-recoverable errors.

Sticky safety intent

--complete-product only ratchets up. If you opt into the longer Ralph chain on first run, restarts honor it even when the operator forgets to re-pass the flag. The reverse — silent downgrade on resume — would be a contract violation.

Runtime selection ≠ phase delegation

The --runtime flag picks the LLM backend, but it does not hand the whole pipeline to that backend as one big subagent task. INTERVIEW and SEED_GENERATION run in-process; only RUN actually dispatches out. The auto-runtime-semantics.md doc exists specifically because users keep expecting otherwise.
